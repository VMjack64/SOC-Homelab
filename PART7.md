## Log Analysis Warmup

I first made a failed connection to the Windows instance on purpose, then actually connected to the Windows instance, and in it, I created a text file named “hello-world” with the text containing, “Hello World!”. The intent is to create two event logs: A failed brute-force authentication event log, and a successful authentication event log that I could use to correlate with the created text file event log. MyDFIR talks a little more about brute force attacks in the Day 11 video, but for a run down, a brute force attack is where an attacker tries guessing the correct password to an account, whether it’s via pure guessing (akin to trying 0001, 0002, 0003, and so on on a safe’s password input) or a list of words/numbers or common passwords from previous successful breaches (dictionary attack). A credential stuffing attack, on the other hand, uses credentials from a successful breach at one company to breach an account used at another company. Credential stuffing attacks differ from brute force attacks in that a username & password combo is used.

Basically, by adding both the secure and unsecure firewalls to the Windows Server instance after finishing the telemetry gathering session, what AWS is actually doing is combining the rules of both firewalls into one firewall; in other words, the Windows Instance now has one big firewall with the inbound rules of both individual firewalls:

Initially, I assumed AWS firewalls worked in a layered approach, where if multiple firewalls are involved, certain traffic can pass through one firewall, but gets blocked by the other, a concept referred to as defense in depth, which I became familiar with from the Google Cybersecurity course & studying for my Security+ certification. Though when looking back at my course notes, it seems that I missed a few pointers regarding using firewalls in a defense in depth strategy:

Here, a firewall is used to filter traffic from the internet to a zone called the demilitarized zone (DMZ), which acts like an airport security checkpoint to the organization’s internal network zone, with another firewall in between the two to filter out even more traffic. Map of Failed Authentications
Now for the dashboard. I’m going to first populate it with a couple of maps visualizing where the attacks are coming from. According to the documentation about generating a choropleth map, the following pre-components are required:
Data that can provide real-world coordinates (such as the IP addresses from the authentication events).
A lookup table file, which defines regions & its boundaries, such as the boundaries of each U.S. state.
A geospatial lookup, which matches coordinate data to defined regions from the lookup table file.
For the last two, Splunk Enterprise thankfully has built-in files for the U.S. (lookup file: ‘geo_us_states’), as well as the U.S. + other countries (lookup file: ‘geo_countries’), which is enough for the job.
